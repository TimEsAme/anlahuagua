## 1.baseurl 和pathname 区别

baseURL 就是一个网站的基础地址，包括协议、域名和端口，比如 `https://www.xxx.com:8080`,pathname 就是这个网站后面的路径，比如 `/user/list`,完整的 URL 就是由 baseURL、pathname，再加上查询参数和 hash 组成的。



## 2.页面首次加载白屏怎么排查？ 

先看 Network,确认 JS、CSS 有没有加载失败, 再看 Console 有没有报错,因为很多白屏都是 JS 执行异常导致的, 然后检查 Vue 有没有正常挂载 , 再看看是不是路由死循环或者接口一直阻塞, 如果都没问题,就考虑是不是首屏资源太大,导致加载时间过长。



## 3.二次封装组件库的思路 

思路是**在不影响原有使用方式的前提下进行增强**,使用 **`defineProps` 接收原来的属性,通过 `$attrs` 透传没有声明的属性,再把 `slots` 和 `expose` 继续暴露出去**,这样开发者使用的时候和原来的基本没有区别,只是多了一些额外能力,这样既降低了业务代码的重复,又不会影响组件原有的功能和扩展性。



## 4.ECharts 地图常用配置有哪些？ 

- `tooltip`：悬浮提示

- `visualMap`：颜色映射/数据分段

- `geo`：地图坐标系（注册地图数据、缩放、中心点）

- `series`：

  - `type: 'map'`
  - `map`：注册的地图名称
  - `data`：数据数组
  - `roam`：是否允许缩放平移
  - `label`：是否显示地名
  - `itemStyle`：区域颜色、边框

  

##  5.ECharts 怎么重新渲染？ 

- **数据变化**：`myChart.setOption(newOption)`（增量合并，自动更新）
- **完全重置**：`myChart.setOption(newOption, true)`（第二个参数 `true` 表示不合并）
- **容器大小变化**：`myChart.resize()`
- **销毁重建**：`myChart.dispose()` → `echarts.init(dom)`



## 6.项目里很多下拉框都需要请求数据，怎么控制？ 

- **统一请求管理**：封装一个 `getSelectData` 函数，内部做**请求去重**（同一接口同时只发一个请求，其他等待结果）
- **缓存策略**：请求成功后存入内存/缓存，下次直接取缓存
- **按需加载**：下拉展开时才触发请求（`@focus` / `@visible-change`）
- **防抖**：搜索型下拉加防抖控制



## 7.call、apply、bind 的区别 

区别就是：call立即执行,参数一个一个传,apply立即执行,参数放数组里. bind不会立即执行,而是返回一个新的函数



## 8.Vue2 组件通信有哪些方式？ 

父子组件一般就是 props 和 $emit,层级比较深可以用 provide/inject.兄弟组件以前经常用EventBus,全局共享数据一般会用 Vuex.还有一些特殊情况也可以用$ref $parent、$children,不过项目里现在基本不用了。

provide 一个**函数**或 **computed**，每次获取时调用函数取最新值

##  11.箭头函数与普通函数的区别

箭头函数最大的特点就是没有自己的 this,它会继承外层作用域的 this.另外它不能作为构造函数,没有 arguments,也不能通过call、apply、bind去修改this。



## 12.什么是异步传染性?怎么消除异步的传染性？ 

当一个函数依赖异步操作（Promise、await）时，它本身也必须以异步的方式返回结果（通常返回 Promise），导致调用它的函数也需要处理异步，最终异步会沿着调用链不断向上传播。

Webpack 5 在生产模式下默认内置了 **SplitChunksPlugin**、**TerserPlugin** 和 **Tree Shaking**，只需配置即可生效，无需额外安装插件。核心优化策略包括：通过 **SplitChunksPlugin** 将 node_modules 抽离为独立的 vendors 文件以利用浏览器缓存，配合 **import()** 实现路由与组件的动态懒加载；借助 **Tree Shaking** 配合 package.json 的 sideEffects: false 剔除未引用的 ES Module 代码；使用 **TerserPlugin** 压缩 JS，**CssMinimizerPlugin** 压缩 CSS，并用 **MiniCssExtractPlugin** 将 CSS 提取为独立文件；通过 **ImageMinimizerPlugin** 压缩图片资源，**HtmlWebpackPlugin** 开启 minify 删除注释、空格及冗余属性，同时压缩内联的 CSS 和 JS；组件库则通过 **babel-plugin-import** 实现按需引入，避免全量打包；最后配合 **CompressionWebpackPlugin** 开启 Gzip 压缩，进一步减小网络传输体积。整体上围绕 **压缩、摇树、拆分、按需** 四个方向，从代码体积、加载效率和传输成本三方面系统性地缩小打包产物。



## 14.Webpack 打包原理 

Webpack 会从入口文件开始递归分析依赖，构建依赖图，通过 Loader 转换模块、Plugin 扩展能力，最后打包生成 Bundle，因此开发时需要先完成一次打包，项目越大启动越慢。 





## 15.Vite 打包原理 

Vite 开发环境利用浏览器支持的ES Module ，不需要提前打包，浏览器请求哪个模块就编译哪个模块，同时使用 esbuild 预构建第三方依赖，所以启动速度和热更新都更快。到了生产环境，Vite 会使用 Rollup 进行打包，实现 Tree Shaking、代码分割等优化。 



## 16.esbuild 预构建第三方依赖 

Vite 在开发环境中使用 esbuild 对第三方依赖进行预构建，主要是将多个分散的模块合并成少量文件，并将 CommonJS 模块转换为 ESM 格式。这样可以减少浏览器请求数量，同时解决模块兼容性问题，从而提升开发环境的启动速度和加载性能。 



## 17.到了生产环境，Vite 会使用 Rollup 进行打包

Vite 在开发环境使用 esbuild 做依赖预构建以提升启动速度，而在生产环境会使用 Rollup 进行打包，因为 Rollup 在 Tree Shaking、代码分割和打包优化方面更成熟，能够生成更小、更高质量的生产代码。 



## 18.RESTful风格api

RESTful 风格：**GET 请求主要使用 query 参数**，因为它用于查询数据，参数放在 URL 中，便于分享、缓存和分页筛选；**POST、PUT、PATCH 请求主要使用 body**，因为需要提交数据，请求体没有 URL 长度限制，也更适合传输复杂对象；而**资源唯一标识（如用户 ID、商品 ID）通常放在 params 中**，例如 /users/:id，用于定位具体资源。
